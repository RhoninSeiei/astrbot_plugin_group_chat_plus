# `matoi_guardian_ep5_cn` 角色扮演群聊插件设计

## 1. 目标

基于现有 `astrbot_plugin_group_chat_plus` 主线，拆分出一个**独立插件**，仅服务少量指定群，且只以 `matoi_guardian_ep5_cn` 这一人格参与群聊。

新插件需要满足以下目标：

1. 与现有 `chat_plus` 同实例并存，互不影响。
2. 保留当前主线中已经验证过的群聊判断骨架，包括读空气、回复策略、主动发言、重复拦截、LivingMemory 联动等。
3. 删除与角色扮演关系较弱的功能，降低维护面。
4. 强制群白名单，只有显式填写的群号才会启用。
5. 决策、回复、主动发言、记忆读写统一绑定 `matoi_guardian_ep5_cn`，不依赖 AstrBot 当前默认人格。
6. 为长期开发、维护和上下文压缩提供独立的实施记录文档。

## 2. 非目标

本项目首版不处理以下事项：

1. 支持多个角色扮演人格。
2. 自动与旧插件协商群号归属。
3. 兼容私聊链路。
4. 保留 Web 管理面板和其相关安全、鉴权、日志系统。
5. 把现有通用能力先抽成共享引擎再做新插件。

## 3. 总体方案

采用“**从现有插件直接分叉，保留群聊主骨架，按角色扮演场景做减法与定制**”的方式。

新插件是一个独立私有仓库，不再作为原仓库的同步分支维护。初始代码来自当前 `chat_plus` 主线快照，但后续视为单独项目。

该方案的重点不是重构抽象层，而是优先保证以下三件事：

1. 安装层面可以与旧插件并存。
2. 行为层面可以稳定继承现有群聊主线能力。
3. 角色扮演差异可以明确收敛在少数专用模块中。

## 4. 仓库与插件边界

### 4.1 独立仓库

新插件使用独立私有 GitHub 仓库，建议仓库名采用人格标识，例如：

- `astrbot_plugin_matoi_guardian_ep5_cn`

仓库层面不建立长期 fork 关系。后续维护以新仓库为准。

### 4.2 独立插件身份

以下标识全部换新：

1. 插件目录名
2. Python 包名
3. `metadata.yaml` 中的 `name`、`display_name`、`desc`、`help`、`repo`
4. 配置文件名
5. 日志前缀
6. 指令前缀

目标是让 AstrBot 在同一实例中同时加载旧插件与新插件时，配置、日志、命令和数据都能明确区分。

### 4.3 独立数据与缓存

新插件必须拥有独立的数据目录和状态数据，包括：

1. 上下文缓存
2. 未回复消息缓存
3. 主动发言状态
4. 图片描述缓存
5. 回复历史缓存
6. 记忆命名空间

两个插件之间不能共享任何插件私有缓存。

## 5. 服务范围与硬约束

### 5.1 强制群白名单

新插件改为强制白名单模式，规则如下：

1. 白名单为空时，插件整体视为未启用。
2. 收到群消息后，第一步先检查群号是否命中白名单。
3. 白名单外群聊一律跳过，不进入缓存、回复、记忆、主动发言链路。
4. 主动发言仅允许在白名单群内运行。

旧插件保持现状，由配置层人工避免群号重叠。

### 5.2 固定人格绑定

插件内部定义唯一人格标识：

- `matoi_guardian_ep5_cn`

决策、回复、主动发言、记忆注入统一强制写入这一人格，不读取 AstrBot 当前默认人格。

如果当前 AstrBot 环境中找不到该人格，插件进入“已加载但停用”状态，仅记录清晰日志，不处理任何消息。

## 6. 功能保留、改写与移除

### 6.1 保留的主干能力

以下能力保留，并作为首版骨架：

1. 群消息入口
2. 上下文缓存与平台历史读取
3. 读空气判断
4. 回复概率与时段调整
5. 回复生成链路
6. 重复回复拦截
7. 回复密度限制
8. 对话疲劳
9. 主动发言调度
10. LivingMemory 联动
11. 发送前内容清洗与兜底保护

### 6.2 需要改写为角色扮演版本的能力

以下模块保留结构，但改成角色扮演语境：

1. 决策提示词
2. 回复提示词
3. 主动发言提示词
4. 记忆注入提示
5. 消息质量判断的语义倾向
6. 情绪、注意力、插话意愿相关提示

核心原则是：判断标准从“通用群聊机器人如何参与”改为“`matoi_guardian_ep5_cn` 在该群情境下是否会自然参与，以及应当如何参与”。

### 6.3 移除的能力

以下内容从新仓库中删除：

1. `web/` 管理面板
2. 私聊模块
3. 与私聊处理相关的缓存和工具
4. 面向通用用户的管理型说明内容
5. 与角色扮演弱相关的面板安全逻辑

## 7. 角色扮演场景下的消息处理策略

### 7.1 消息判断

消息判断保留“两层过滤”结构：

1. 概率层控制整体频率
2. AI 判断层控制角色一致性

AI 判断应围绕三件事：

1. 当前消息是否属于角色会注意到的内容。
2. 当前情境下角色是否适合出声。
3. 如果出声，是否应以角色语气回应。

`@`、明确点名、直接向角色提问的消息仍保持高优先级；纯水聊、无关话题、与角色身份弱关联的话题倾向于降权。

### 7.2 回复生成

回复生成继续沿用“上下文整理、记忆注入、提示词拼装、模型请求、结果清洗”的链路，但统一围绕角色身份约束。

回复层需要明确限制：

1. 禁止退回通用助手口吻。
2. 禁止解释系统规则或内部机制。
3. 禁止越出该人格的认知范围与关系边界。
4. 保持对群成员称呼、距离感和语气的一致性。

### 7.3 主动发言

主动发言保留，但触发条件收紧为角色扮演版本。

主动发言前至少叠加以下限制：

1. 群在强制白名单中。
2. 近期话题与角色存在合理关联，或群氛围适合角色插话。
3. 连续主动发言次数和无人接话次数受更严格限制。
4. 发言前进行一次“角色是否会在此刻主动出声”的 AI 判断。

主动发言的目标从“活跃群聊”调整为“维持角色存在感”，可以是短句评论、旁观式插话或轻量互动，而不是强行开新话题。

### 7.4 记忆联动

LivingMemory 联动继续保留，但只服务这一人格和这一组白名单群。

记忆的主要内容应当偏向：

1. 群成员关系
2. 长期话题
3. 群内约定
4. 角色过往态度
5. 影响角色表达连续性的长期背景

## 8. 代码结构与模块组织

### 8.1 顶层文件

保留以下顶层骨架：

1. `main.py`
2. `_conf_schema.json`
3. `metadata.yaml`
4. `README.md`
5. `CHANGELOG.md`
6. `utils/`
7. `docs/`

删除：

1. `web/`
2. `private_chat/`

### 8.2 新增角色扮演专用模块

建议在 `utils/` 下新增以下模块：

1. `persona_binding.py`
   负责固定人格的选择、校验和请求注入。

2. `roleplay_group_guard.py`
   负责强制白名单校验，以及白名单为空时的停用判定。

3. `roleplay_prompt_policy.py`
   负责角色扮演场景下的决策提示、回复提示、主动发言提示组装。

4. `roleplay_memory_namespace.py`
   负责 LivingMemory 命名空间隔离与角色视角记忆读写标识。

### 8.3 命名原则

命名采用“旧主干保留，新增差异模块显式命名”的原则：

1. 现有稳定模块名尽量保留。
2. 所有角色扮演专用代码使用 `roleplay_` 或 `persona_` 前缀。
3. 日志前缀统一换新。
4. 指令前缀统一换新。

## 9. 配置策略

配置采用混合方案：

### 9.1 尽量沿用的配置

核心行为参数优先保留现有键名和数值语义，例如：

1. 回复概率
2. 时段调整
3. 注意力机制
4. 对话疲劳
5. 回复密度
6. 记忆注入
7. 主动发言相关阈值

### 9.2 必须换新的配置

以下内容改成新命名：

1. 插件名与显示名
2. 配置文件名
3. 日志标识
4. 指令前缀
5. 仓库信息

### 9.3 新增的角色扮演专用配置

新增配置用于描述角色扮演场景，例如：

1. 角色扮演专用主动发言提示词
2. 角色一致性约束
3. 群内关系描述
4. 禁止越界表达的规则补充
5. 世界观或设定补充提示

## 10. 错误处理与兼容规则

### 10.1 前置条件不满足

以下情况属于“不能处理”，只记录日志并跳过：

1. 固定人格不存在
2. 白名单为空
3. 当前群不在白名单中
4. 配置声明依赖的记忆能力不可用

### 10.2 运行时失败

以下情况属于“处理失败”，按局部降级思路处理：

1. 模型请求超时
2. 主动发言判断失败
3. 记忆检索失败
4. 提示词构建异常

要求单个模块异常不能拖垮整条消息处理链。

### 10.3 与旧插件并存的兼容规则

并存规则只有三条：

1. 新旧插件可同时安装。
2. 群号作用范围由配置人工分开。
3. 配置、数据目录、命名空间、日志标识、命令前缀完全独立。

## 11. 迁移实施顺序

建议分四步推进：

1. 从当前 `chat_plus` 主线复制出新仓库初始版本。
2. 完成插件身份隔离与非相关模块删除。
3. 接入固定人格、强制群白名单、独立命名空间。
4. 改写角色扮演相关提示词、主动发言条件和文档。

这个顺序便于逐步验证，出现问题时更容易定位。

## 12. 验证策略

验证分四层：

### 12.1 装载验证

确认新旧插件能够在同一 AstrBot 实例中被同时发现和加载。

### 12.2 边界验证

确认白名单群内只有新插件生效，白名单外群一律跳过；旧插件在其原有群中行为保持原状。

### 12.3 人格一致性验证

确认决策、回复、主动发言、记忆读写都强制绑定 `matoi_guardian_ep5_cn`。

### 12.4 稳定性验证

确认重复拦截、记忆隔离、主动发言失败抑制、超时降级、脏输出清洗仍然有效。

## 13. 文档与长期实施记录

除本设计文档外，项目还需要一份长期维护用的实施记录文档：

- `docs/superpowers/records/matoi-guardian-ep5-plugin-implementation-log.md`

该文档持续记录：

1. 当前阶段目标
2. 已完成事项
3. 关键决策
4. 影响行为的配置变更
5. 验证结果
6. 已知问题
7. 下一阶段计划
8. 面向上下文压缩的阶段摘要

设计文档负责回答“要做什么”；实施记录负责回答“做到哪一步、为什么这样做、后续从哪里接着做”。

## 14. 成功标准

满足以下条件即可视为首版设计目标达成：

1. 新插件以独立身份存在于独立私有仓库。
2. 与旧插件可同实例并存。
3. 只在强制白名单群内工作。
4. 全链路固定使用 `matoi_guardian_ep5_cn`。
5. 保留现有群聊主线稳定性保障。
6. 删除 Web 与私聊等非核心能力。
7. 具备长期实施记录文档，便于维护和上下文压缩。
